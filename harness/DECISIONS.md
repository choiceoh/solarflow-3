# SolarFlow 설계 판단 기록

모든 주요 설계 판단의 "왜?"를 기록합니다.
새 대화창에서 "왜 이렇게 했지?"라는 질문이 나오면 이 문서를 참조합니다.

## D-001: Go+Rust 아키텍처 분리
- **결정**: Go(프론트+API) + Rust(계산엔진)
- **이유**: Go는 변경이 잦은 UI/API에 적합, Rust는 복잡한 계산(Landed Cost, 재고집계)에 안정적
- **분리 기준**: 한 행 사칙연산=Go, 여러 테이블 조합=Rust
- **날짜**: 2026-03

## D-002: 엑셀 기반 입력 방식 채택
- **결정**: SolarFlow = 입력 시스템 → 엑셀 Import/Export 허브 + 분석 시스템
- **이유**: (1) SolarFlow 다운 시 엑셀 원본이 실무자 PC에 있어 업무 마비 없음 (2) 디원/화신은 ERP 없어서 SolarFlow가 유일한 시스템, 다운 리스크 큼 (3) 실무자 업무 방식 변경 최소화
- **날짜**: 2026-03-29

## D-003: ToggleStatusRequest를 common.go에 배치
- **결정**: model/company.go가 아닌 model/common.go에 배치
- **이유**: is_active 토글은 company, manufacturer, product 등 여러 핸들러에서 공통 사용
- **날짜**: 2026-03-29

## D-004: 감리자-시공자 분리 도입
- **결정**: 코드 작성(시공자)과 설계 검증(감리자)을 별도 대화창으로 분리
- **이유**: 시공자(Claude)가 3번의 대화창에서 반복적으로 "Go만으로 충분" 하며 설계를 임의 변경. 감리자가 설계문서 대비 코드를 대조하여 이 패턴을 차단.
- **날짜**: 2026-03-28

## D-005: Validate()가 error가 아닌 string 반환
- **결정**: Validate() 메서드가 Go 관례(error)가 아닌 string 반환
- **이유**: 시공자의 초기 구현. 동작에 문제없고 모든 핸들러에 일관 적용됨. 전체 변경 비용 대비 이득이 적어 유지.
- **날짜**: 2026-03-29

## D-006: 자동 검증 3층 구조 도입
- **결정**: (1층) go build+vet+test+CI+린터 (2층) 감리자 대화창 (3층) 상사 코드 리뷰(미확보)
- **이유**: 감리자 수동 검증의 한계. 기계가 잡을 수 있는 것은 기계에게 맡기고, 감리자는 설계 방향 판단에 집중.
- **날짜**: 2026-03-29

## D-007: 엑셀 양식 기반 입력 + 서버 검증
- **결정**: 실무자가 SolarFlow에서 빈 양식 다운로드 → PC에서 입력 → 업로드 시 서버 검증
- **이유**: (1) 엑셀에 드롭다운(제조사, 창고 등) 넣으면 입력 오류 감소 (2) 업로드 시 행별 에러 반환으로 잘못된 데이터 차단
- **날짜**: 2026-03-29

## D-008: receipt_matches.outbound_id FK 지연 추가 [완료]
- **결정**: outbound 테이블이 아직 없으므로 UUID 컬럼만 생성, FK는 Step 9 이후 ALTER TABLE로 추가
- **이유**: 테이블 생성 순서 의존성. receipts/receipt_matches는 Step 8, outbound는 Step 9.
- **상태**: Step 9에서 outbound 테이블 생성 완료. FK 추가 가능 상태.
- **날짜**: 2026-03-29

## D-009: 수금 매칭 합계 검증 Go 허용
- **결정**: receipt_matches의 SUM(matched_amount) ≤ receipt.amount 체크를 Go에서 수행
- **이유**: 두 테이블 조회이지만 단순 합산. Go+Rust 분리 기준의 "한 행 사칙연산" 범위 내.
- **날짜**: 2026-03-29

## D-010: deposit_rate 0~100 범위 검증 추가
- **결정**: 수주의 계약금 비율(deposit_rate)에 0~100 범위 검증 추가
- **이유**: 시공자 자체 판단으로 추가. 계약금 비율이 100%를 넘는 건 실무에서 불가능.
- **날짜**: 2026-03-29

## D-011: limit_changes에 Update/Delete 없음
- **결정**: 한도 변경이력 테이블에 수정/삭제 API를 제공하지 않음 (List, Create만)
- **이유**: 한도 변경은 이력이므로 수정/삭제하면 감사 추적 불가. 잘못 입력 시 새 이력으로 정정.
- **날짜**: 2026-03-29

## D-012: omitempty 일괄 적용
- **결정**: 19개 UpdateRequest 구조체의 모든 포인터 필드에 `json:",omitempty"` 태그 일괄 적용
- **이유**: UpdateRequest의 포인터 필드에 omitempty가 없으면 null 필드도 DB에 전송되어 의도치 않은 덮어쓰기 가능. 부분 업데이트 시 전송하지 않은 필드가 null로 초기화되는 문제 방지.
- **날짜**: 2026-03-29

## D-013: 출고 취소 3단계 (active/cancel_pending/cancelled)
- **결정**: 출고 상태를 active → cancel_pending → cancelled 3단계로 관리
- **이유**: 실무자가 취소 여부를 즉시 결정 못할 수 있음. cancel_pending은 가용재고 미차감. 실무자가 판단 후 삭제 또는 복원.
- **날짜**: 2026-03-29

## D-014: ERP 관리구분과 usage_category 일치
- **결정**: 출고 usage_category를 ERP 관리구분 기반으로 재설계 (기존 7개 → 9개)
- **이유**: ERP 내보내기 시 매핑 정확성. ERP 1,881건 분석 기반으로 재설계. replacement는 construction_damage로 대체. repowering은 outbounds에서 제거, orders.management_category에는 유지.
- **날짜**: 2026-03-29

## D-015: 수주 충당 소스 구분 (fulfillment_source: stock/incoming)
- **결정**: 수주에 fulfillment_source 컬럼 추가 (stock/incoming)
- **이유**: 미착품에도 판매/공사 예약이 걸릴 수 있음. "총확보량"이 실제보다 부풀려지는 것을 방지. 실무자가 "이 수주는 미착품에서 충당"이라는 의사결정을 기록.
- **날짜**: 2026-03-29

## D-016: Axum 선택
- **결정**: Rust 웹 프레임워크로 Axum 채택
- **이유**: SolarFlow 계산엔진은 극한 성능이 아닌 정확한 계산이 목적. Tokio 팀 제작, 타입 안전, 메모리 효율 최고, 학습 곡선 완만. sqlx/serde 등 Tokio 생태계와 자연스럽게 통합.
- **날짜**: 2026-03-29

## D-017: Supabase Session pooler 사용
- **결정**: Supabase Session pooler(포트 5432)를 사용하여 sqlx로 연결
- **이유**: Direct connection은 IPv6 전용이라 fly.io에서 연결 불가. Session pooler(포트 5432)는 IPv4 지원하며 prepared statements도 정상 동작.
- **날짜**: 2026-03-29

## D-018: Cargo.lock 커밋
- **결정**: Cargo.lock을 .gitignore에 넣지 않고 커밋
- **이유**: Rust 공식 권장 — 바이너리 프로젝트는 Cargo.lock을 커밋해야 빌드 재현성(reproducibility) 보장.
- **날짜**: 2026-03-29

## D-019: /health와 /health/ready 분리
- **결정**: 헬스체크를 /health(서버 생존)와 /health/ready(DB 연결)로 분리
- **이유**: /health는 fly.io 헬스체크용 (항상 200, DB 무관). /health/ready는 DB 연결 확인용 (Go에서 Rust 호출 전 상태 확인). DB 장애 시 서버는 살아있지만 계산은 불가능한 상태를 구분.
- **날짜**: 2026-03-29

## D-020: Go↔Rust 통신 패턴
- **결정**: Go에서 Rust가 다운되어도 CRUD 기능 유지 (graceful degradation)
- **이유**: Rust 엔진은 계산 전용이므로, 다운 시 계산 기능만 비활성화. 모든 Rust 호출은 EngineClient.CallCalc()을 통해 일관되게 수행. fly.io auto_stop 시 콜드 스타트 1~3초, 타임아웃 10초로 대응.
- **날짜**: 2026-03-29

## D-021: 재고 집계를 단일 API로 제공
- **결정**: 물리적/가용/미착품을 한 번의 POST /api/calc/inventory 호출로 일괄 계산
- **이유**: 별도 API로 분리하면 시점 차이로 데이터 불일치. 8단계 계산이 서로 의존하므로 한 번에 실행해야 정합성 보장.
- **날짜**: 2026-03-29

## D-022: 장기재고 판별은 최초 입고일 기준 (FIFO 미적용)
- **결정**: 장기재고를 최초 입고일(MIN(actual_arrival)) 기준으로 판별 (normal/warning/critical)
- **이유**: 정확한 FIFO는 건별 추적 필요. 현재는 품번 단위 집계이므로 최초 입고일이 실용적. 실무 문제 시 추후 확장.
- **날짜**: 2026-03-29

## D-023: 부대비용 배분 기준은 capacity_kw 비율
- **결정**: 부대비용을 B/L 라인아이템의 capacity_kw 비율로 배분
- **이유**: 물리적 처리 비용(운송, 보관, 하역)은 용량에 비례. 수량 기준 배분은 Wp이 다른 모듈 혼합 시 부정확.
- **날짜**: 2026-03-29

## D-024: 현재 환율은 최근 면장 환율 사용 (실시간 API 미연동)
- **결정**: 환율 비교 시 가장 최근 면장의 exchange_rate를 현재 환율로 사용
- **이유**: 외부 의존성 최소화. 실시간 환율 필요 시 Phase 확장.
- **날짜**: 2026-03-29

## D-025: Landed Cost 계산 결과는 save=true일 때만 DB 저장
- **결정**: 계산 결과를 save=false(기본값)로 미리보기, save=true일 때만 cost_details 업데이트
- **이유**: Rust의 직접 WRITE는 Go CRUD 통제권과 충돌. 미리보기 후 확인하고 저장하는 것이 실무에 자연스러움.
- **날짜**: 2026-03-29

## D-026: allocated_expenses를 동적 맵으로 처리
- **결정**: 부대비용 배분 내역을 HashMap<String, f64> / map[string]float64로 처리 (키 하드코딩 없음)
- **이유**: expense_type이 추가되어도 코드 변경 불필요. 확장성.
- **날짜**: 2026-03-29

## D-027: LC 수수료 환율은 B/L 환율 우선, 없으면 최근 면장 환율
- **결정**: LC 수수료 계산 시 환율을 B/L 환율 → 최근 면장 환율 순으로 조회
- **이유**: LC에 연결된 B/L이 있으면 해당 B/L의 환율이 가장 정확. 없으면 최근 면장 환율로 대체.
- **날짜**: 2026-03-29

## D-028: 한도 복원 타임라인은 maturity_date 기준
- **결정**: LC 한도 복원을 settlement_date가 아닌 maturity_date 기준으로 계산
- **이유**: settlement_date는 실제 결제 후에만 기록됨. 계획 단계에서는 maturity_date가 유일한 기준.
- **날짜**: 2026-03-29

## D-029: 만기 알림 severity (0~3일 critical, 4~7일 warning)
- **결정**: LC 만기 알림을 0~3일 critical, 4~7일 warning으로 분류
- **이유**: 3일 이내는 즉시 조치 필요, 4~7일은 사전 준비 구간.
- **날짜**: 2026-03-29

## D-030: LC 수수료는 자동 계산(예상). 수동 입력은 Phase 확장.
- **결정**: 현재는 요율 기반 자동 계산만. 실제 은행 청구 금액 수동 입력은 Phase 확장 (LC 수수료 자동 계산은 완료, 수동 보정만 미구현).
- **이유**: 1원 단위 차이는 마진에 영향 없으나 정확성 위해 수동 보정 기능 예정.
- **날짜**: 2026-03-29

## D-031: 마진 원가는 품번별 가중평균 (FIFO 미적용)
- **결정**: 마진 분석 시 원가를 품번별 가중평균(입고 수량 기준)으로 계산
- **이유**: FIFO는 건별 추적 필요. 가중평균이 실무에서 충분하고 구현 간결.
- **날짜**: 2026-03-29

## D-032: cost_basis 선택 (cif/landed)
- **결정**: 마진 분석에서 원가 기준을 cif(회계용)/landed(실무용) 중 선택 가능
- **이유**: 회계팀은 CIF 기준 보고, 판매팀은 부대비용 포함 Landed 기준 의사결정. 한 API에서 둘 다 지원.
- **날짜**: 2026-03-29

## D-033: 미수금 경과일 기준은 출고일 (outbound_date)
- **결정**: 미수금 경과일을 세금계산서 발행일이 아닌 출고일 기준으로 계산
- **이유**: 세금계산서 미발행 건도 포함해야 미수금 전체 파악 가능. 출고일이 더 보수적.
- **날짜**: 2026-03-29

## D-034: 거래처 분석에 마진율+계약금비율 포함
- **결정**: 거래처별 분석에 avg_margin_rate, avg_deposit_rate 포함
- **이유**: "미수금 많은 업체 → 마진은? → 계약금은?" 한 화면에서 판단 가능.
- **날짜**: 2026-03-29

## D-035: 수급 전망 입고예정은 B/L ETA 기준
- **결정**: 입고예정을 PO 계약일이 아닌 B/L ETA 기준으로 월별 배분
- **이유**: PO 계약일은 실제 입고와 수개월 차이. B/L 미생성분은 unscheduled로 분리.
- **날짜**: 2026-03-29

## D-036: 수급 전망 출고예정은 delivery_due 기준
- **결정**: 출고예정을 delivery_due(납기 요청일) 기준으로 월별 배분
- **이유**: 납기 요청일이 실무 출고 시점. NULL은 unscheduled로 분리.
- **날짜**: 2026-03-29

## D-037: fulfillment_source='incoming' 수주는 수급 전망에서 제외
- **결정**: 미착품 충당 수주는 월별 재고 흐름 계산에서 제외
- **이유**: 미착품 충당 수주는 물리적 재고에 영향 안 줌. 포함하면 이중 차감.
- **날짜**: 2026-03-29

## D-038: closing 음수 허용 (insufficient 플래그)
- **결정**: 기말재고가 음수가 되어도 0으로 자르지 않고 insufficient=true 표시
- **이유**: 실무자 사전 경고. 0으로 자르면 이후 달 예측 부정확.
- **날짜**: 2026-03-29

## D-039: 수금 매칭 추천은 미수금 합계가 입금액을 초과하지 않는 조합만 제시
- **결정**: 과매칭 방지 — 조합 합계 > 입금액인 경우 제외
- **이유**: 과매칭은 실무 발생 안 함. 차액은 선수금/이월 (Phase 4).
- **날짜**: 2026-03-29

## D-040: 부분집합 탐색 N<=20 완전 탐색, N>20 greedy
- **결정**: 미수금 N<=20이면 비트마스크 완전 탐색(2^20 약 100만), N>20이면 greedy
- **이유**: N=20이면 1초 이내. 20건 초과는 극히 드묾. greedy는 날짜 오래된 순으로 실무 관행과 일치.
- **날짜**: 2026-03-29

## D-041: 미수금 status 기준 Step 16과 동일 (30/60일)
- **결정**: 미수금 목록의 status를 거래처 분석과 동일 기준(30/60일)으로 적용
- **이유**: 거래처 분석과 미수금 목록의 status 일관성 유지.
- **날짜**: 2026-03-29

## D-042: 수금 매칭은 현재 outbound_id 기준

- **환경변수**: SUPABASE_URL에서 JWKS URL 자동 유도 ({URL}/auth/v1/.well-known/jwks.json). 또는 SUPABASE_JWKS_URL 직접 지정.

## D-083: 입고완료는 DB에 completed 저장, 화면 표시는 유형별로 분기
- **결정**: 모든 입고유형(해외직수입/국내구매/그룹내구매)의 "입고완료" 상태는 DB에 `status='completed'`로 저장. 화면 표시만 유형에 따라 다르게 (해외직수입: "완료", 국내/그룹: "입고완료"). 국내/그룹은 scheduled→completed 2단계만 사용("입고중"→"입고완료"로 표시).
- **재고 반영 기준**: `shipping/arrived/customs` → 미착품, `completed/erp_done` → 물리적 재고. Rust `engine/src/calc/inventory.rs`의 `fetch_bl_incoming`이 BL 상태 직접 기반으로 집계(PO 매칭 미사용, 유형 무관).
- **이유**: DB 스키마를 단일 status enum으로 유지하여 재고 집계 SQL을 단순화. 화면 표시만 분기해서 UX 혼동 제거.
- **날짜**: 2026-04-07

## D-084: 사이드바 메뉴 3그룹 — 업무흐름/현황분석/도구
- **결정**: 사이드바 메뉴를 업무 수행 순서대로 3그룹으로 재편.
  1. **업무 흐름**: 발주/결제 → 입고관리 → 면장/원가 → 수주/수금 → 출고/판매
  2. **현황/분석**: 재고현황, 은행/LC, 대시보드
  3. **도구**: 마스터관리, 검색, 메모, 결재안, 설정
- **이유**: 기존 "운영/기타" 2그룹 구조가 실제 업무 흐름(좌→우 진행)과 일치하지 않아 신규 사용자 학습 비용이 높았음. 업무 순서대로 나열하여 "다음 단계는 무엇"이 직관적으로 보이도록 재배치.
- **날짜**: 2026-04-07

## D-085: PO → 입고 → 면장 데이터 연결 + 독립 입력 경로 병존
- **결정**: 업무 흐름 간 데이터 이동(자동채움)을 표준화하되 독립 입력 경로도 유지.
  - **PO → 입고**: PO 상세에 "입고 등록" 버튼. 클릭 시 `/inbound?po=xxx`로 이동 → 입고 폼이 열리면서 PO의 법인/제조사/인코텀즈/결제조건/라인 품번·단가 자동 채움. 수량은 비움(분할선적 대비). 모든 필드 수정 가능.
  - **입고 → 면장**: 입고 상세에 "면장 등록" 버튼(해외직수입만). `/customs?bl=xxx`로 이동 → 면장 폼이 B/L 자동 선택 상태로 열림. B/L:면장 = 1:1 원칙. 향후 분할통관 대비 수량 수정 가능.
  - **독립 입력**: 입고관리 "새로 등록"은 기존대로 유지. 입고 폼 최상단 "P/O 연결 (선택사항)" 드롭다운으로 PO 매칭/미매칭 선택 가능. 과거 데이터, 긴급 입고용.
  - **단가 수정 가능**: PO 단가는 기본값일 뿐이며 실무자가 자유롭게 수정. B/L 라인의 실단가가 원가 산정의 정본.
- **이유**: 실무 흐름(PO 작성 → 선적 → 입고 → 통관)이 선형적이지만 과거 데이터 이관/긴급건도 자주 발생. 자동채움으로 일상 업무 생산성을 높이되, 독립 입력 경로를 막지 않아 예외 상황에도 대응 가능하도록 병존.
- **날짜**: 2026-04-07

## D-086: PO 필수 연결 + 분할선적 + 환율 3단계 비교
- **결정**: 모든 입고는 PO에 연결 필수 (가상 PO라도 생성). 분할선적은 동일 PO에 BL 여러 건. 환율은 PO(발주) / 면장(통관) / TT·LC(송금) 3단계 각각 기록하여 비교 가능하게 함. 입고등록 시 PO 선택 → 제조사/품목/단가 자동 채움.
- **이유**: BL번호가 공식 추적키, PO번호는 내부 그룹핑 도구. 기존 데이터는 가상 PO로 묶음. 환율 3단계 비교로 환율 손익 가시화.
- **날짜**: 2026-04-08
