# SolarFlow 운영 이관 설계

작성일: 2026-04-26

## 목적

검증용 데이터를 모두 정리한 뒤 SolarFlow를 실제 운영 기준 시스템으로 전환한다.
전환 이후 실무자는 SolarFlow에서 업무 데이터를 만들고, SolarFlow에서 내려받은 아마란스 입고/출고/매출 자료를 아마란스에 업로드한다.

핵심 원칙은 다음과 같다.

1. SolarFlow에 실제 운영 데이터로 들어올 자료는 입고/출고/매출 엑셀 양식을 사용한다.
2. 아마란스 업로드용 입고/출고 양식과 SolarFlow 입력 양식은 같은 업무 단위를 표현해야 한다.
3. 과거 참고자료는 운영 재고, 수주, 출고, 매출 테이블에 섞지 않는다.
4. 기초재고와 미착품은 운영 데이터이므로 SolarFlow에 실제 데이터로 이관한다.
5. 검증용 데이터 삭제는 백업과 dry-run 집계 후 한 번에 수행한다.

## 데이터 분류

### 1. 운영 마스터

운영 전환 후에도 유지하거나 정비해야 하는 기준정보다.

- 법인
- 제조사
- 품목
- 거래처
- 창고
- 현장
- 은행
- 사용자/권한

마스터는 거래 데이터보다 먼저 정비한다. 품목코드, 제조사명, 규격, 출력, 모듈 크기가 흔들리면 대시보드, 수급전망, 아마란스 내보내기가 모두 흔들린다.

### 2. 운영 이관 데이터

전환일 이후 재고, 미착, 수주, 출고, 판매에 영향을 주는 데이터다.

- 이미 입고되었고 전환일 이후 판매 또는 공사사용될 재고
- SolarFlow 구축 전에 발주되었고 아직 입고 중이거나 입고 예정인 물량
- 전환일 이후 출고될 판매/공사 수주
- 전환일 이후 발행할 세금계산서와 수금/수금매칭

이 데이터는 SolarFlow의 실제 업무 테이블에 입력한다.

### 3. 과거 참고 데이터

2025-01부터 2026-03까지의 과거 거래처별 매출, 이익, 이익률, 제조사별 판매량, 구매량, 상품판매, 공사사용 자료다.

이 데이터는 참고 분석용으로 필요하지만 운영 재고나 수급전망의 계산 소스가 되면 안 된다.
따라서 별도 `legacy` 영역으로 보관하거나, 참고 화면에서만 조회한다.

## 엑셀 파일 사용 원칙

### 입고

SolarFlow에 들어올 입고 데이터는 입고 엑셀 양식으로 입력한다.
아마란스 입고등록 양식과 동일한 업무 단위로 관리한다.

운영 이관 시 입고는 두 종류로 나눈다.

| 구분 | 의미 | SolarFlow 처리 |
| --- | --- | --- |
| 기초재고 | 이미 입고되었고 전환일 현재 창고에 남아 있는 재고 | 입고 업로드 후 B/L 상태를 `completed` 또는 `erp_done`으로 보정 |
| 미착품 | 발주/선적/입항/통관 중이며 아직 재고가 아닌 물량 | 입고 업로드 후 실제 단계에 맞게 `scheduled`, `shipping`, `arrived`, `customs` 유지 |

기초재고는 예시처럼 이관 전용 번호를 부여한다.

- PO: `MIG-PO-20260401-001`
- LC: `MIG-LC-20260401-001`
- B/L: `MIG-BL-20260401-001`

PO나 LC가 없는 과거 발주 건도 임의 번호를 부여해서 추적키를 만든다.
임의 번호는 "없는 자료를 꾸미는 것"이 아니라 SolarFlow에서 추적 가능한 기준키를 만드는 것이다.

### 출고

SolarFlow에 들어올 출고 데이터는 출고 엑셀 양식으로 입력한다.
아마란스 출고등록 양식과 동일한 업무 단위로 관리한다.

출고는 반드시 다음 중 하나로 구분한다.

- 상품판매
- 공사사용
- 무상스페어
- 유지보수
- 폐기/조정
- 그룹내 이동

수주가 있는 출고는 수주번호를 연결한다. 수주 없이 과거 기초재고를 정리하는 출고는 이관 메모를 남긴다.

### 매출/세금계산서

매출 데이터는 출고 이후 입력한다.
현재 SolarFlow 매출 업로드는 출고 ID를 기준으로 연결하므로, 매출을 먼저 넣지 않는다.

표준 순서는 다음과 같다.

1. 수주 등록
2. 출고 등록
3. 매출/세금계산서 등록
4. 수금 등록
5. 수금매칭

## 현재 시스템에서 보정이 필요한 지점

### 입고 업로드 상태값

현재 입고 업로드 API는 B/L 상태를 기본 `scheduled`로 생성한다.
따라서 기초재고를 입고 양식으로 업로드하면 그대로는 가용 재고가 아니라 미착/예정 물량처럼 보일 수 있다.

운영 이관 전 다음 중 하나를 구현해야 한다.

1. 입고 엑셀 양식에 `status` 컬럼을 추가해 `completed`, `erp_done`을 직접 지정한다.
2. 입고 업로드 화면에 "기초재고로 등록" 옵션을 추가해 업로드된 B/L을 `completed`로 생성한다.
3. 업로드 후 선택한 B/L을 일괄 `completed` 또는 `erp_done`으로 바꾸는 이관 보정 도구를 만든다.

권장안은 2번이다. 실무자는 상태 코드를 직접 알 필요가 없고, 이관용 작업인지 일반 입고 예정인지 화면에서 선택하면 된다.

### 과거 참고자료 저장 위치

과거 참고자료는 활성 업무 테이블에 넣지 않는다.
다음 중 하나로 별도 관리한다.

1. `legacy_sales_summary`, `legacy_purchase_summary`, `legacy_inventory_flow` 같은 별도 테이블 생성
2. 엑셀 원본을 보관하고 SolarFlow에서 읽기 전용 참고 화면 제공
3. 대시보드/분석 화면에 "과거 참고" 탭을 별도로 제공

권장안은 1번과 3번의 조합이다.
운영 데이터와 과거 참고 데이터를 DB 레벨에서 분리하면서, 사용자는 분석 화면에서 함께 참고할 수 있다.

## 운영 이관 절차

### 1단계. 전환일 확정

전환일을 하나 정한다.
예: 2026-04-01 또는 실제 운영 시작일.

전환일은 다음 기준이 된다.

- 전환일 현재 남은 재고 = 기초재고
- 전환일 이후 들어올 물량 = 미착품 또는 신규 입고
- 전환일 이후 나갈 물량 = 수주/출고/판매
- 전환일 이전 실적 = 과거 참고자료

### 2단계. DB 백업

검증용 데이터를 삭제하기 전에 전체 DB를 백업한다.
삭제 스크립트는 반드시 dry-run으로 테이블별 삭제 예정 건수를 먼저 출력한다.

### 3단계. 마스터 정비

다음 순서로 정비한다.

1. 법인
2. 제조사
3. 품목
4. 거래처
5. 창고
6. 현장
7. 은행

품목은 제조사, 출력, 규격, 모듈 크기까지 확인한다.
대시보드의 제조사*출력, 모듈크기*출력 집계가 이 마스터에 의존한다.

### 4단계. 검증용 거래 데이터 삭제

삭제 대상은 거래성 데이터다.
마스터는 별도 판단 없이는 삭제하지 않는다.

삭제 후보:

- 발주/PO 라인
- LC
- T/T
- B/L 입고와 입고 라인
- 면장/부대비용
- 수주
- 출고와 B/L 연결
- 매출/세금계산서
- 수금
- 수금매칭
- 재고 배정
- 테스트용 메모/알림

실행 전 실제 테이블명 기준으로 dry-run 집계를 확정한다.

### 5단계. 기초재고 입력

전환일 현재 남은 실제 재고를 입고 엑셀 양식으로 입력한다.

필수 기준:

- 법인
- 제조사
- 품목
- 수량
- 창고
- 원가 기준
- 이관용 B/L 번호
- 상태: `completed` 또는 `erp_done`

기초재고는 수급전망에서 현재 재고로 잡혀야 한다.

### 6단계. 미착품 입력

아직 입고되지 않은 물량을 입고 엑셀 양식으로 입력한다.

필수 기준:

- 임의 또는 실제 PO 번호
- 임의 또는 실제 LC 번호
- 실제 또는 임의 B/L 번호
- ETA
- 제조사/품목/수량
- 현재 단계
- 창고 예정지

미착품은 수급전망에서 입고 예정으로 잡혀야 한다.

### 7단계. 수주/출고/판매 입력

전환일 이후 처리할 업무는 정상 흐름으로 입력한다.

1. 판매예약 또는 수주 등록
2. 출고 등록
3. 매출/세금계산서 등록
4. 수금 등록
5. 수금매칭

과거 출고/판매를 재현하기 위한 입력은 하지 않는다.
단, 전환일 이후 수금될 미수금은 별도 이관 대상이다.

### 8단계. 검증

운영 시작 전 다음을 확인한다.

- 가용재고의 품목별 수량
- 수급전망의 기초재고와 미착품 월별 반영
- 수주 잔량
- 출고 후 실제 재고 차감
- 매출과 세금계산서 발행 상태
- 수금/수금매칭 잔액
- 아마란스 입고 내보내기
- 아마란스 출고 내보내기

## 권장 작업 순서

1. 입고 업로드에 "기초재고 등록" 옵션 추가
2. 검증용 거래 데이터 삭제 dry-run 스크립트 작성
3. 과거 참고자료용 `legacy` 저장 구조 설계
4. 이관용 입고/출고 엑셀 템플릿 점검
5. 마스터 정비
6. DB 백업
7. 검증 데이터 삭제
8. 기초재고 업로드
9. 미착품 업로드
10. 수급전망/대시보드 검증
11. 아마란스 내보내기 검증

## 결론

엑셀 파일에서 SolarFlow로 실제 들어올 데이터는 입고와 출고 엑셀 파일에 입력해 SolarFlow에 업로드하는 방식이 맞다.
다만 과거 참고자료는 같은 방식으로 운영 테이블에 넣으면 안 된다.

운영 이관에서 가장 중요한 구분은 다음 한 줄이다.

> 앞으로 재고, 출고, 매출, 수금에 영향을 줄 데이터만 SolarFlow 운영 데이터로 넣고, 과거 실적은 별도 참고 데이터로 분리한다.
