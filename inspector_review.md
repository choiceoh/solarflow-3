## 1. 핵심 원칙 (Alex 확정)

### 원칙 1: PO가 모든 데이터의 출발점
- PO 없는 입고는 없다. 가상 PO라도 반드시 생성한다.
- PO 번호는 내부 관리 번호이며, 공식 번호가 아니다 (B/L, 면장번호와 다름).
- 모든 하위 데이터(TT, LC, BL, 면장, 출고)는 PO에서 정보를 끌어온다.

### 원칙 2: 재고는 "모델 기준"으로 본다
- "진코 640W 몇 MW 있지?" — 여러 PO/BL에서 온 동일 모델을 합산.
- 재고 현황/가용재고는 제조사 → 규격(Wp) → 모델명 기준으로 조회.

### 원칙 3: 이익분석은 "BL 기준"으로만 가능하다
- 출고가 발생해야 → BL이 특정되고 → 해당 BL의 면장 원가가 확정되어야 → 이익 계산 가능.
- 가용재고 화면에서는 이익분석 불가 (원가 정보가 없으므로).
- BL별 출고추적 → 출고마감 → 매출마감이 순서.

### 원칙 4: 입고/출고 모두 본품+스페어 구분 필요
- 입고: 본품(유상) + 스페어(무상) — 현재 BL 라인아이템에 반영됨.
- 출고: 본품(유상 판매) + 스페어(무상 출고) — 스페어 출고분도 판매원가에 포함하여 정확한 이익 계산.

## 2. 전체 데이터 흐름 (확정)

```
① PO 등록 (발주)
   제조사, 품목(본품+스페어), 단가(USD/Wp), 총수량(MW), 
   Incoterms, 결제조건, 계약유형(스팟/프레임), 발주환율①
   ↓
② T/T 계약금 송금 (PO 총액의 5~10%)
   분할 가능 (1차, 2차...), 추가 계약금(단가 변동분)도 발생 가능
   송금환율③ 기록
   ↓
③ L/C 개설 (잔금)
   PO에서 분할 개설 (100MW → 10MW × 10건 등)
   LC 한도 사전 확인/증액/만기 복원 모니터링 필요
   스팟도 대규모면 분할 LC
   ↓
④ 선적 → B/L 발행 (LC당 N건 분할선적)
   입고등록 시 PO 선택 → 제조사/품목/단가 자동 채움
   BL마다 다른 것만 입력: BL번호, ETD/ETA, 수량, 환율
   ↓
⑤ 통관 → 면장 발행 (BL당 대부분 1건)
   면장환율② 확정 → CIF Landed 원가 확정
   부대비용 등록 → Landed Cost 계산
   ↓
⑥ 재고 반영
   [모델 기준 — 가용재고 확인용]
   물리적 재고 = 입고완료 합계 - 출고완료 합계
   가용재고 = 물리적 - 예약(재고) - 배정
   미착품 = 선적중~통관전 물량
   가용미착품 = 미착품 - 예약(미착품)
   총확보량 = 가용재고 + 가용미착품
   ↓
⑦ 출고
   [BL 기준 — 출고추적/이익분석용]
   
   (a) 상품 판매:
       거래처, 판매가(원/Wp), 본품 수량, 스페어 수량(무상)
       → FIFO로 어느 BL에서 나가는지 결정
       → 해당 BL 면장 원가로 이익 계산
       → 스페어 무상분도 판매원가에 포함 (매출=0, 원가=있음)
       → BL 전량 출고 시 출고마감 → 매출마감
   
   (b) 공사 사용:
       현장명, 수량
       → 이익분석 없음, 사용처 추적만
   ↓
⑧ 분석/대시보드
   PO 기준: 계약량, LC개설량, 입고완료, 미착품, 잔량
   BL 기준: 입고수량 vs 출고수량, 출고마감 여부
   거래처별: 매출 집계, 이익분석, 월별 추이
   환율 3단계 비교: 발주환율① vs 면장환율② vs 송금환율③
```

### 3.2 🔴 GAP — 반드시 보강 필요 (6건)

#### GAP-1: 입고등록 시 PO 선택 → 자동채움 [최우선]

**현재**: 입고등록 폼에서 PO를 선택하는 UI가 없음. BL 테이블에 po_id 컬럼은 존재하나, 프론트에서 PO 선택 → 관련 정보 자동 채움 기능이 없음.

**보강안**:
1. 입고등록 폼 상단에 "PO 선택" 드롭다운 추가 (PO번호 검색 가능)
2. PO 선택 시 자동으로 채워지는 필드:
   - 제조사 (manufacturer_id)
   - 법인 (company_id)
   - 통화 (currency)
   - Incoterms
   - 결제조건
3. PO의 발주품목(po_line_items) 목록을 BL 라인아이템 초기값으로 제시
4. 사용자는 BL 고유 정보만 입력: BL번호, ETD, ETA, 실제입항일, 항구, 수량(분할분), 환율
5. 같은 PO에서 BL 여러 개 생성 가능 (분할선적)

**DB 변경**: 없음 (bl_shipments.po_id 이미 존재)
**Go 변경**: GET /api/v1/purchase-orders/{id}/line-items 응답에 자동채움 필드 포함 확인
**프론트 변경**: 입고등록 폼에 PO 선택 컴포넌트 + 자동채움 로직 추가

#### GAP-2: PO 발주품목 ↔ BL 품목 수량 추적 [높음]

**현재**: po_line_items 테이블과 bl_line_items 테이블 사이에 직접 연결(FK)이 없음. PO에 "진코 640W 5,000장"을 등록해도, BL에서 몇 장이 선적되었는지 PO 품목 단위로 추적 불가.

**보강안**:
- bl_line_items에 `po_line_id UUID(FK)` 컬럼 추가 (nullable, 국내구매 등 PO 없는 경우)
- BL 생성 시 PO 품목 선택 → 해당 품목의 잔여량에서 차감
- PO 상세에서 품목별 진행률 표시: 계약수량 / 선적완료 / 잔여
- 계산: `잔여 = PO품목수량 - SUM(해당 po_line_id를 참조하는 BL 라인아이템 수량)`

**DB 변경**: `ALTER TABLE bl_line_items ADD COLUMN po_line_id UUID REFERENCES po_line_items(po_line_id);`

#### GAP-3: BL 출고추적 + 출고마감 상태 [최우선]

**현재**: BL 상태가 입고 과정만 추적 (scheduled → shipping → arrived → customs → completed → erp_done). 입고 완료 이후 해당 BL의 물량이 전부 출고되었는지 추적하는 상태가 없음.

**보강안**:
- BL 상태에 출고 추적 상태 추가:
  ```
  입고 과정: scheduled → shipping → arrived → customs → completed → erp_done
  출고 과정: outbound_partial (일부 출고) → outbound_closed (전량 출고) → sales_closed (매출마감)
  ```
- 출고 발생 시 해당 BL의 출고 합계 자동 갱신
- BL 입고수량 = BL 출고수량 일치 시 → outbound_closed 전환
- 매출마감 완료 시 → sales_closed 전환

**의미**: 포워더/창고 청구서가 BL 단위로 발행되므로, BL별 출고 완료 확인이 실무적으로 필수. 이 상태가 있어야 "이 BL은 다 처리됐다"를 시스템에서 확인 가능.

#### GAP-4: 출고 FIFO 원가 매칭 (출고 → BL 원가 연결) [높음]

**현재**: D-022, D-031로 미해결. outbounds 테이블에 `bl_line_id`(어느 BL에서 나간 물량인지) FK가 없음. 이익분석 시 출고 건의 원가를 역추적할 수 없음.

**보강안**:
- outbounds 테이블에 `bl_line_id UUID(FK)` 컬럼 추가
- 출고 등록 시 FIFO 자동 매칭:
  1. 해당 품번(product_id)의 BL 라인아이템을 입고일(actual_arrival) 순으로 정렬
  2. 가장 오래된 BL부터 잔여수량 확인
  3. 출고수량을 BL 잔여수량에서 차감
  4. 한 BL로 부족하면 다음 BL로 넘어감 (한 출고가 여러 BL에 걸칠 수 있음)
- 출고 1건이 여러 BL에 걸치는 경우 → outbound_fifo_details 매핑 테이블 필요:
  ```
  outbound_fifo_id | UUID | PK
  outbound_id      | UUID(FK) | 출고 연결
  bl_line_id       | UUID(FK) | BL 라인아이템 연결
  quantity          | INTEGER  | 해당 BL에서 나간 수량
  unit_cost_krw_wp  | DECIMAL  | 해당 BL의 면장 원가(원/Wp)
  ```
- 이익분석: `SUM(판매수량 × 판매가) - SUM(FIFO수량 × BL면장원가)` = 이익

**FIFO 계산 주체**: Rust (여러 테이블 조합 — D-001 기준)

#### GAP-5: 출고에 본품/스페어 구분 추가 [높음]

**현재**: outbounds 테이블에 `spare_qty` (스페어 수량) 필드만 존재. 스페어가 별도 라인아이템이 아니라 단순 숫자임. 스페어 출고의 원가가 이익분석에 포함되지 않음.

**보강안**:
- outbounds 테이블에 `item_type VARCHAR(10)` 추가: main(본품) / spare(스페어)
- 또는 기존 spare_qty 활용하되, 이익분석 시 스페어 원가 포함 로직 추가:
  ```
  판매 이익 = 판매매출(본품+스페어) - 판매원가(본품+스페어)
  
  본품: 매출 = 수량 × 판매가   /  원가 = 수량 × BL면장원가
  스페어: 매출 = 0 (무상)       /  원가 = 수량 × BL면장원가  ← 여기가 핵심
  
  총 이익 = 본품매출 - (본품원가 + 스페어원가)
  ```
- 스페어 원가가 판매원가에 포함되어야 "실제로 얼마 남았는지" 정확한 이익 계산 가능

**권장안**: outbound_fifo_details (GAP-4)에서 본품/스페어를 모두 추적하므로, FIFO 매핑 시 스페어 수량도 별도 행으로 기록. 이익분석에서 자동 합산.

#### GAP-6: 메뉴 구조 재배치 [중간]

**현재 사이드바**:
```
재고 현황
대시보드
─────────
입고 관리
발주/결제
출고/판매
수주/수금
면장/원가
은행/LC
마스터 관리
─────────
검색
메모
결재안
─────────
설정
```

**문제**: 은행/LC가 발주/결제 아래에 떨어져 있음. 실무에서 LC 개설은 PO 직후 단계. 면장/원가는 입고 이후 단계인데 별도 메뉴.

**보강안**:
```
[업무흐름] — 실제 업무 순서
  ① 발주/결제 (PO + T/T)
  ② 은행/LC (LC 개설 + 한도관리)
```

## 4. 상세 설계: 재고현황

### 4.1 가용재고 (모델 기준 합산)

**조회 화면**: 제조사 선택 → 모델별 목록

| 제조사 | 모델명 | 규격 | 크기(mm) | 물리재고 | 예약 | 배정 | 가용재고 | 미착품 | 미착예약 | 가용미착 | 총확보 |
|--------|--------|------|----------|---------|------|------|---------|--------|---------|---------|--------|
| 진코 | JKM640N | 640Wp | 2465×1134 | 9,000장(5.76MW) | 2,000 | 500 | 6,500 | 3,000 | 1,000 | 2,000 | 8,500 |
| 트리나 | TSM-720N | 720Wp | 2384×1303 | 4,500장(3.24MW) | 1,000 | 0 | 3,500 | 5,000 | 2,000 | 3,000 | 6,500 |

**집계 공식** (기존 확정, 변경 없음):
```
물리적 재고 = SUM(입고 completed/erp_done 수량) - SUM(출고 active 수량)
예약 = SUM(fulfillment_source='stock' AND sale/spare/maintenance/other 수주잔량)
배정 = SUM(fulfillment_source='stock' AND construction/repowering 수주잔량)
가용재고 = 물리적 - 예약 - 배정
미착품 = PO(contracted/shipping) 잔량 - 해당 PO 입고완료량
미착품예약 = SUM(fulfillment_source='incoming' 수주잔량)
가용미착품 = 미착품 - 미착품예약
총확보량 = 가용재고 + 가용미착품
```

**핵심**: 여기서는 원가 정보가 없음. "몇 개 있는지"만 확인하는 뷰.

### 4.2 재고 상세 (BL 출처 드릴다운)

가용재고 행 클릭 시 → 해당 모델의 BL별 상세:

| BL번호 | 입고일 | 면장환율 | CIF원가(원/Wp) | 입고수량 | 출고수량 | 잔여 | 상태 |
|--------|--------|---------|--------------|---------|---------|------|------|
| JWSH26010014 | 2026-01-15 | 1,432 | 145.2 | 3,000 | 3,000 | 0 | 출고완료 |
| JWSH26020021 | 2026-02-20 | 1,445 | 148.1 | 3,500 | 1,500 | 2,000 | 일부출고 |
| JWSH26030008 | 2026-03-10 | 1,438 | 142.5 | 2,500 | 0 | 2,500 | 미출고 |

이 뷰에서 각 BL을 클릭하면 → 출고추적 상세로 이동.

## 5. 상세 설계: 출고추적 (BL 기준)

### 5.1 BL별 출고 현황

### 5.2 이익분석 (BL 단위)

```
BL-JWSH26010014 이익분석:
  
  [판매 건]
  바로 1,000장(본품) × 170원/Wp × 0.640kW = 108,800,000원 (매출)
  바로 10장(스페어) × 0원 = 0원 (무상, 매출 없음)
  신명 1,500장(본품) × 168원/Wp × 0.640kW = 161,280,000원
  신명 15장(스페어) × 0원 = 0원
  
  판매 매출 합계: 270,080,000원
  
  [판매 원가 — 본품+스페어 전부 포함]
  본품 2,500장 × 145.2원/Wp × 0.640kW = 232,320,000원
  스페어 25장 × 145.2원/Wp × 0.640kW = 2,323,200원
  
  판매 원가 합계: 234,643,200원
  
  [공사 사용 — 이익분석 없음, 원가만 기록]
  공사 500장(본품) × 145.2원/Wp × 0.640kW = 46,464,000원
  공사 5장(스페어) × 145.2원/Wp × 0.640kW = 464,640원
  
  ──────────────
  판매 이익: 270,080,000 - 234,643,200 = 35,436,800원
  판매 이익률: 13.1%
  공사 사용 비용: 46,928,640원 (별도 표기)
```

### 5.3 스페어(무상) 출고의 이익분석 포함 원리

입고 시 스페어가 무상(free)이더라도, 실제로는:
- 해당 BL의 운송비, 통관비, 부대비용에 스페어 물량도 포함됨
- 면장 CIF 원가에 스페어 인보이스 금액이 반영됨 (무상도 인보이스 금액 있음 — 현재 설계에 bl_line_items.invoice_amount_usd "무상도 금액 있음" 명시)
- 따라서 스페어를 판매 시 무상으로 내보내면, **매출은 0인데 원가는 발생** → 이익 감소 요인

이것을 이익분석에서 빠뜨리면 "실제 남은 돈"이 과대 계산됨.

## 7. DB 변경 사항 요약

### 7.1 새 테이블 (1개)

```sql
CREATE TABLE outbound_fifo_details (
  outbound_fifo_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  outbound_id UUID NOT NULL REFERENCES outbounds(outbound_id),
  bl_line_id UUID NOT NULL REFERENCES bl_line_items(bl_line_id),
  quantity INTEGER NOT NULL,           -- 해당 BL에서 나간 수량
  item_type VARCHAR(10) NOT NULL,      -- main/spare
  unit_cost_krw_wp DECIMAL(10,2),      -- 면장 CIF 원가(원/Wp)
  landed_cost_krw_wp DECIMAL(10,2),    -- Landed 원가(원/Wp)
  created_at TIMESTAMP DEFAULT now()
);
```

### 7.2 컬럼 추가 (2개)

```sql
-- BL 라인아이템 ↔ PO 라인아이템 연결
ALTER TABLE bl_line_items 
ADD COLUMN po_line_id UUID REFERENCES po_line_items(po_line_id);
```

## 10. 설계 결정 제안 (감리 승인 요청)

### D-087: PO 기준 전체 데이터 자동 연결
- **결정**: PO가 모든 데이터의 출발점. 입고등록 시 PO 선택 필수 → 데이터 자동 채움.
- **이유**: 수동 입력 최소화, 데이터 정합성 보장, PO→LC→BL→면장→출고 전체 추적 가능.

### D-088: BL 출고마감 상태 추가
- **결정**: BL 상태에 outbound_partial, outbound_closed, sales_closed 추가.
- **이유**: 포워더/창고 청구서가 BL 단위. BL별 전량 출고 확인이 실무 필수.

### D-089: FIFO 원가 매칭 (outbound_fifo_details)
- **결정**: 출고 시 FIFO로 BL 원가 자동 매칭. outbound_fifo_details 테이블 신규.
- **이유**: 이익분석의 정확성. 기존 D-022, D-031 해결.

### D-090: 스페어(무상) 출고 원가 포함
- **결정**: 스페어 무상 출고분의 원가를 판매원가에 포함하여 이익 계산.
- **이유**: 매출=0이지만 원가 발생. 빠뜨리면 이익 과대 계산.

### D-091: 메뉴 구조 업무흐름 기반 재배치
- **결정**: 사이드바를 업무흐름/분석/도구 3그룹으로 재배치.
- **이유**: 실무자가 위→아래 순서로 업무 수행 가능.
